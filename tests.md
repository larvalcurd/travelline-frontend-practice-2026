Вот готовый список тест-кейсов и сценариев, составленный строго под архитектуру вашего проекта (с учетом связки CurrencyConverterPage, ConverterCard, CurrencyInputRow и RateDivider), который вы можете делегировать для написания кода.
------------------------------
## Что именно нужно протестировать (Объекты тестирования)

   1. Интеграционный уровень (CurrencyConverterPage): Взаимодействие состояния (useState), расчетных формул (useMemo) и ваших мок-данных из папки src/mocks.
   2. Компонентный уровень (ConverterCard и дочерние элементы): Корректность передачи пропсов, обработка событий ввода/клика и изоляция DOM-дерева через key.

------------------------------
## Сценарии для тестирования (Test Cases)## Сценарий 1: Корректность первоначального рендера (Initial State)

* Цель: Проверить, что мок-данные правильно подставились в UI при первой загрузке.
* Что проверять:
* В первом селекте выбрана базовая валюта (например, PLN).
   * Во втором селекте выбрана дефолтная целевая валюта (например, JPY).
   * В текстовом поле суммы (Amount) отображается дефолтное значение 1.
   * В заголовке PairSummary выводится строка вида 1 Polish Zloty is.
   * В поле результата отображается актуальный базовый курс из моков (например, 38.45 Japanese Yen).

## Сценарий 2: Реактивный пересчет при изменении суммы

* Цель: Проверить математическую логику конвертации на UI.
* Что проверять:
* При изменении значения в инпуте Amount (например, с 1 на 2) срабатывает событие onAmountChange.
   * Значение во втором (readonly) инпуте автоматически пересчитывается по формуле сумма * курс (в нашем примере должно стать 76.90).
   * Дробная часть результата всегда округляется до двух знаков после запятой (.toFixed(2)).

## Сценарий 3: Запрет на выбор одинаковых валют (Валидация бизнес-логики)

* Цель: Проверить, что приложение не позволяет пользователю сломать конвертер выбором пары USD/USD.
* Что проверять:
* Если в первом селекте выбрана валюта PLN, а пользователь пытается выбрать во втором селекте тоже PLN, приложение перехватывает это событие.
   * Первый селект остается PLN, а второй селект автоматически переключается на любую другую доступную валюту из моков (например, USD или JPY).
   * Проверить обратную ситуацию: при выборе в первом селекте той же валюты, что стоит во втором, автоматически переключается первый.

## Сценарий 4: Работа кнопки Swap (Смена направления)

* Цель: Проверить корректность обмена валют местами.
* Что проверять:
* До клика: первый селект = PLN, второй = JPY.
   * Пользователь нажимает на кнопку ⇄ Swap.
   * После клика: первый селект становится JPY, второй — PLN.
   * Сумма результата в нижнем инпуте мгновенно пересчитывается по новому обратному курсу из объекта MOCK_PRICE_CHANGES['JPY']['PLN'].

## Сценарий 5: Сброс дочернего состояния по key (Reset by Key)

* Цель: Убедиться, что компонент MoreAboutPair полностью пересоздается, а не переиспользуется.
* Что проверять:
* При изменении валютной пары в селектах (например, с PLN/JPY на USD/EUR), строка pairLabel меняется.
   * Так как pairLabel используется в качестве key для <MoreAboutPair key={data.pairLabel} />, старый DOM-узел компонента должен полностью удалиться из дерева (Unmount).
   * Новый компонент должен смонтироваться (Mount) с обновленным текстовым заголовком More about USD/EUR и новыми блоками описания (USD и EUR).

## Сценарий 6: Вывод динамического контента по макету

* Цель: Проверить конкатенацию строк в блоке подробностей.
* Что проверять:
* Заголовки внутри MoreAboutPair отображаются строго в формате Название - Код - Символ (например: Japanese Yen - JPY - ¥).
   * Если в моках у валюты отсутствует описание (description), на экране должен отобразиться дефолтный fallback-текст: No description available for this currency..

## Каким образом это технически тестировать?
Человеку или ИИ, которому вы поручите писать код, нужно передать следующие вводные по стеку:

   1. Стек: Vitest + React Testing Library + jsdom.
   2. Тип тестов: Интеграционные тесты (рендерить весь компонент CurrencyConverterPage).
   3. Метод поиска элементов: Использовать строго семантические запросы RTL (screen.getByLabelText для инпутов, screen.getAllByRole('combobox') для селектов, screen.getByRole('button') для кнопок).
   4. Метод симуляции: Использовать fireEvent.change() для ввода текста/выбора опций и fireEvent.click() для нажатия на Swap и разделитель.

Этот список полностью покрывает требования критериев приемки вашего домашнего задания.
